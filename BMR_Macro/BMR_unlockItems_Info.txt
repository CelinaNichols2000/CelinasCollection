  SECTION 3 — HOW THE INANIMATE SYSTEM WORKS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  CLIENT SIDE:
    GAME_MANAGER.instance.inanimate
      → READ-ONLY getter: () => !!_character.item
      → Cannot be set directly
      → To make it return false: set gm.character.item = null
        (gm.character returns _character by reference, mutation works)

    gm.Send(action, payload)
      → Has a CLIENT-SIDE inanimate guard
      → Blocks Crafting, Location, and other actions when inanimate
      → Bypass: use raw WebSocket directly (skip gm.Send entirely)

  RAW WEBSOCKET ACCESS:
    The WebSocket is stored under a Symbol-keyed property on GAME_MANAGER.
    Symbol description = 'ws'

    function getWS() {
      const sym = Object.getOwnPropertySymbols(GAME_MANAGER)
        .find(s => s && s.description === 'ws');
      if (sym && GAME_MANAGER[sym] && GAME_MANAGER[sym].readyState === 1)
        return GAME_MANAGER[sym];
    }

    Sending directly on this WS bypasses gm.Send and the inanimate guard.
    BUT: the server has its own inanimate guard — see Section 4.

  SERVER SIDE:
    The server checks character state before processing ANY action.
    If your character is inanimate (has item assigned), the server rejects:
      - Crafting actions (all optionIds)
      - Location changes
      - Most Form actions EXCEPT escape
    This is a server-side guard — raw WS tricks cannot bypass it.

  NPC UNLOCK (the correct unlock mechanic):
    Located at: ward location
    Crafting optionId: 29  (NPC nurse unlock — removes permanent + frees item)
    Crafting optionId: 27  (alternative/fallback remove-permanence)
    Requirement: sender must be the OWNER of the item
    Flow: Location{ward} → Crafting{itemId, optionId:29, npc:true}
    This was DESIGNED for the owner to use on their item, not for the item
    to use on itself.

  ESCAPE MECHANIC (designed for inactive owner):
    Form action: { action: 'Form', escape: true }
    Requirements:
      - sealed = false  ✓ (our item is not sealed)
      - permanent = true is a potential blocker (unclear)
      - owner must have been offline for N days (server checks this)
    Result if successful: character becomes a "stray" unowned item
    After escape: ANY player can NPC-unlock the stray at the ward
    Status: TRIED — did not work. Exact server response unknown.
